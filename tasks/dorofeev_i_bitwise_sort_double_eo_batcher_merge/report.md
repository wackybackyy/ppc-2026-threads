## 1. Introduction

Данная работа посвящена реализации параллельного алгоритма поразрядной сортировки (Radix Sort) для чисел с плавающей
точкой двойной точности (`double`) с последующим объединением отсортированных блоков с помощью алгоритма
четно-нечетного слияния Бэтчера (Odd-Even Batcher Merge). Ожидаемый результат — создание корректно работающих
последовательной (SEQ) и параллельных (OMP, TBB, STL) версий алгоритма, а также анализ их производительности.

## 2. Problem Statement

**Задача:** Отсортировать массив элементов типа `double`.
**Ограничения:** * Данные типа `double` нельзя напрямую сортировать классической поразрядной сортировкой из-за особенностей
  представления чисел с плавающей точкой в памяти (стандарт IEEE 754).

- Необходимо использовать четно-нечетное слияние Бэтчера, которое требует, чтобы количество сливаемых блоков было
  степенью двойки.

## 3. Baseline Algorithm (Sequential)

Последовательный алгоритм состоит из двух этапов:

1. **Локальная сортировка (LSD Radix Sort):** Массив чисел `double` побитово преобразуется в массив беззнаковых
   целых чисел `uint64_t`. Знаковый бит инвертируется специальным образом, чтобы сохранить естественный порядок чисел
   (отрицательные числа становятся меньше положительных). После этого применяется стандартная 8-проходная (побайтовая)
   сортировка подсчетом (Counting Sort). По завершении числа конвертируются обратно в `double`.
2. **Слияние Бэтчера:** Массив делится на независимые логические блоки. Блоки попарно сливаются с помощью
   итеративного алгоритма Odd-Even Merge.

## 4. Parallelization Scheme

**Декомпозиция (для OMP, TBB, STL):**

- Исходный массив при необходимости дополняется фиктивными элементами (максимальными значениями `double`) до размера,
  являющегося ближайшей степенью двойки.
- Массив разбивается на `N` независимых блоков (чанков), где `N` — наибольшая степень двойки, не превышающая
  количество доступных потоков (PPC_NUM_THREADS).

**Синхронизация и распределение:**

- На первом этапе каждый поток получает свой чанк и выполняет локальную независимую сортировку (Radix Sort).
  Синхронизация не требуется.
- На втором этапе (дерево слияния) на каждом новом уровне размера `size` потокам параллельно раздаются задачи по
  четно-нечетному слиянию пар блоков. В конце каждого уровня происходит барьерная синхронизация (неявная в OMP/TBB
  и через `join()` в STL).

## 5. Implementation Details

- **Обработка чисел IEEE 754:** Для применения целочисленной поразрядной сортировки (Radix Sort) к типу `double`
  выполняется побитовое копирование элементов в тип `uint64_t`. Для сохранения правильного порядка (особенно для
  отрицательных чисел) выполняется битовая манипуляция со знаковым битом: если число отрицательное, инвертируются
  все биты, если положительное — только старший (знаковый) бит.
- **Дополнение массива (Padding):** Алгоритм четно-нечетного слияния Бэтчера структурно требует, чтобы количество
  объединяемых элементов было степенью двойки. В реализации предусмотрено динамическое расширение исходного вектора
  фиктивными элементами (`std::numeric_limits<double>::max()`) до ближайшей степени двойки перед началом работы, с
  последующим усечением до оригинального размера на этапе постобработки.
- **Управление памятью и индексация:** Для минимизации накладных расходов при параллельном доступе к независимым
  подмассивам (чанкам) итерация и передача границ в рабочие потоки осуществляется через арифметику сырых
  указателей (`double* raw_data`). Сама поразрядная сортировка (Counting Sort) реализована не in-place и требует
  выделения временного локального буфера размером `O(N)` внутри каждого потока.
- **Специфика режима ALL:** В версии ALL реализован механизм одновременной инициализации и барьерной синхронизации
  различных парадигм (распределенной памяти MPI и разделяемой памяти OMP, TBB, STL) в едином адресном пространстве
  для валидации отсутствия конфликтов между их внутренними планировщиками потоков.

## 7. Results and Discussion

### 7.1 Correctness

Корректность работы всех реализаций подтверждена прохождением набора функциональных тестов (`ppc_func_tests`) через
Google Test. Результаты параллельных версий побитово совпадали с результатами работы последовательной версии на
различных объемах данных (включая пустые массивы и массивы с отрицательными числами).

### 7.2 Performance

Время измерялось метрикой `task_run` (чистое время работы алгоритма).
Базовое время (Baseline) — время последовательной версии (SEQ) на 1 потоке: **0.01237 s**.

**Анализ результатов:**
Как видно из таблицы, максимальное ускорение (1.86x) достигается на технологии TBB при 8 потоках.
Однако общая эффективность распараллеливания стремительно падает при увеличении числа ядер. Это объясняется крайне
малым объемом тестируемых данных: алгоритм завершает работу за 6–12 миллисекунд. В таких условиях накладные расходы
(overhead) ОС на создание пула потоков, переключение контекста и барьерную синхронизацию сопоставимы со временем
самой полезной работы. При тестировании на массивах большого размера (миллионы элементов) ожидается линейный рост
ускорения.

## 8. Conclusions

Успешно разработаны и протестированы четыре версии алгоритма: последовательная и три параллельные (OMP, TBB, STL).
Алгоритм показал абсолютную корректность работы со спецификой IEEE 754. TBB-версия с использованием
`tbb::task_arena` и `tbb::parallel_for` оказалась наиболее эффективной благодаря продвинутому встроенному
планировщику задач (Task Scheduler), который быстрее справляется с накладными расходами на малых объемах
данных по сравнению с системными вызовами `std::thread` и реализацией OpenMP.
